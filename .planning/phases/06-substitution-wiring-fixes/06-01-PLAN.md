<objective>
Fix the two simulation-side wiring bugs that make pitching changes and fatigue cosmetic. After this plan, the at-bat simulation actually uses (a) the current pitcher recorded in GameState (not the lineup's frozen starting_pitcher_id) and (b) fatigue-modified pitching stats — so substitutions and pitcher tiredness produce measurable changes in outcomes.

**Critical correction from revision:** The TUI's `GameScreen.advance_game` (~line 397-406) calls `self.engine.sim.simulate_at_bat(...)` DIRECTLY and never enters `engine.simulate_half_inning`. The only caller of `simulate_half_inning` outside tests is the orphaned `simulate_game()` (per v1.0-MILESTONE-AUDIT.md). Fixing fatigue inside `simulate_half_inning` ALONE closes the audit gap in test-only code while the user-visible bug stays broken. Therefore this plan extracts a small helper `resolve_pitcher_stats(state, pitching_team)` that performs the pitcher_id lookup + fatigue modification, and calls it from `advance_game` (the real hot path).

**Disposition note on simulate_half_inning:** `simulate_half_inning` keeps its existing `pitching_stats` argument (no signature churn) and applies fatigue inline via `apply_fatigue_modifier(pitching_stats, current_state.current_pitcher_fatigue.current_fatigue)` per iteration. This means the fatigue formula lives in TWO call sites (resolve_pitcher_stats + simulate_half_inning's per-AB loop) and is NOT a single source of truth. This is an intentional tradeoff: signature stability over DRY. The plan summary at output time must call out: "if the fatigue formula (currently hits*=1+f*0.5, walks*=1+f*0.3, HRs*=1+f*0.4) changes, update both call sites." The two regression tests in Task 1 (TestResolvePitcherStats + TestFatigueEffectsSim) cover both code paths so a formula drift breaks at least one test.

Purpose: Close audit gaps 1 and 2 (SUBS-01) at the actual code path the user exercises. Without this, the rest of Phase 6 (UI wiring, DH rules) operates on a sim that ignores its inputs.
Output: New module-level helper `resolve_pitcher_stats`; `GameScreen.advance_game` calls it; `GameEngine.simulate_half_inning` calls `apply_fatigue_modifier` directly per iteration; six regression tests prove both paths.
</objective>
