<objective>
Close the three TUI-side audit gaps (3, 4, 5):
- Gap 3 (SUBS-02): SubstitutionMenu.compose() currently only yields pitcher widgets. The batters list is constructed by GameScreen.show_substitution_menu but never rendered. Render it.
- Gap 5 (SUBS-03): GameScreen._handle_substitution calls sub_manager.record_substitution() directly, bypassing GameEngine.make_substitution()'s validation (no-re-entry, DH forfeiture from Plan 02). Route through the engine.
- Gap 4 (SUBS-03): GameScreen._reset_game() leaves the old SubstitutionManager attached, so removed players carry over into a replayed game. Replace it via an extracted `_reset_sub_manager` helper with unit-test coverage.

Depends on 06-01 (engine simulation correctness so a pinch-hit batter's stats actually drive subsequent at-bats) and 06-02 (make_substitution now sets dh_forfeited correctly, so routing through it gives the right rule behavior).

**Disposition on DH-forfeiture TUI trigger (per revision):** The substitution menu in this plan exposes TWO sub-types only: `pitching_change` (pitcher swap, new_position=Position.PITCHER) and `pinch_hitter` (batter swap, new_position=None). Neither path triggers DH forfeiture through user interaction. The engine-side fix in Plan 02 is therefore unreachable from the TUI in v1.0.

**Chosen option: (a) defer the TUI defensive-replacement selector to a future phase + add an engine-API test that exercises the forfeiting branches directly.** This keeps the `autonomous: no` scope of this plan tight (one human checkpoint, three TUI tasks) instead of adding a third menu selector with its own UX/CSS/keybindings burden. The engine-API test is added in this plan (Task 4) and proves the forfeiture wiring works end-to-end even though no TUI control invokes it yet.

**Other intentional UX changes documented here:** Removal of the auto-pick-first-pitcher fallback in SubstitutionMenu is an intentional UX improvement beyond audit gap 3 — currently the modal commits a substitution to whichever pitcher happens to be first in the list if the user presses Confirm without selecting one. This is removed because it masks selection failures and produces "phantom" substitutions. The new behavior dismisses with None (no substitution) if nothing is selected. Documented in the plan summary at output time.

Purpose: After this plan, SUBS-01/02/03 are end-to-end functional through the TUI (with the DH-forfeiture defensive-replacement UI explicitly deferred).
Output: Substitution modal renders both lists with non-clipping sizing; one validated code path executes subs via extracted is_away_team helper; replay resets state via extracted helper with unit-test coverage; engine-API test proves DH forfeiture reachable via the engine.
